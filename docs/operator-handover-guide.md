# SwingRadar 운영 인수인계 가이드

## 0. 이 문서의 목적

이 문서는 `SwingRadar`를 운영하는 사람이 매일 무엇을 확인해야 하는지, 문제가 생기면 어떤 순서로 판단하고 대응해야 하는지, 사용자 문의가 오면 어디까지 직접 처리하고 어디서 개발자에게 넘겨야 하는지를 한 번에 정리한 운영용 문서입니다.

이 문서는 다음 사람을 기준으로 작성했습니다.

- 개발자가 아닌 운영 담당자
- 서비스를 처음 인수받는 사람
- 매일 서비스 상태를 확인하고 공지, 사용자, 후보 종목, 기록 이슈를 관리해야 하는 사람

이 문서를 다 읽고 나면 아래를 할 수 있어야 합니다.

- 오늘 서비스를 그대로 열어도 되는지 판단할 수 있다.
- `/admin`에서 어디를 먼저 봐야 하는지 안다.
- 추천 품질, 데이터 지연, 배치 실패, 기록 꼬임 같은 문제를 구분할 수 있다.
- 사용자 문의가 왔을 때 바로 안내할 수 있다.
- 운영자가 직접 해결 가능한 일과 개발자에게 넘겨야 할 일을 구분할 수 있다.

## 1. 가장 먼저 알아야 할 한 줄

운영자의 가장 중요한 역할은 `좋은 종목을 더 많이 찾는 것`이 아닙니다.  
운영자의 가장 중요한 역할은 `오늘 서비스가 믿을 만한 상태인지 확인하고, 문제가 있을 때 빠르게 범위를 줄이는 것`입니다.

즉 운영자는 아래 네 가지를 매일 반복합니다.

1. 서비스가 오늘 열려도 되는지 확인한다.
2. 추천과 데이터의 신뢰도를 확인한다.
3. 사용자에게 보여지는 공지와 상태를 정리한다.
4. 계정, 세션, 기록 꼬임, 사용자 문의를 처리한다.

## 2. 서비스 구조를 운영자 관점에서 이해하기

### 2-1. 사용자가 보는 핵심 흐름

사용자는 보통 아래 순서로 서비스를 사용합니다.

1. `Today`에서 오늘 해야 할 일을 본다.
2. `Opening Check`에서 장 시작 전 판단을 한다.
3. `Signals`에서 후보 종목을 본다.
4. `Analysis`에서 한 종목의 진입/손절/목표가를 본다.
5. `Portfolio`에서 실제 보유와 기록을 관리한다.
6. `Reviews`와 `Performance`에서 복기한다.
7. `Account`에서 보안, 가이드, 정책, 테마를 관리한다.

운영자는 이 흐름이 오늘도 끊김 없이 이어지는지만 먼저 보면 됩니다.

### 2-2. 운영자가 보는 핵심 화면

운영자는 아래 화면을 중심으로 일합니다.

| 화면 | 운영 목적 | 매일 보는 정도 |
| --- | --- | --- |
| `/admin` `Overview` | 오늘 서비스 전체 상태를 한눈에 판단 | 매우 높음 |
| `/admin` `Data Quality` | 추천과 데이터 품질 확인 | 매우 높음 |
| `/admin` `Candidate Ops` | 오늘 후보 종목 검토와 승격 | 높음 |
| `/admin` `Users` | 계정, 세션, 정지, 삭제, 지원 이슈 처리 | 필요 시 |
| `/admin` `Notices` | 공지 팝업 등록과 일정 관리 | 필요 시 |
| `/admin` `Portfolio` | 사용자 기록 이슈 대응, 테스트 포트폴리오 조정 | 필요 시 |
| 사용자 화면 직접 확인 | 실제 서비스가 사용자에게 어떻게 보이는지 검증 | 매일 |

### 2-3. 운영자가 직접 관리하는 것과 아닌 것

운영자가 직접 관리하는 것:

- 서비스 상태 확인
- 공지 팝업 관리
- 오늘 후보 종목 검토 상태와 메모
- 사용자 세션 정리
- 사용자 계정 정지/해제
- 사용자 계정 정보 수정
- 사용자 삭제 요청 처리
- 기록 꼬임 신고 접수 후 1차 확인
- 운영 체크포인트 기록

운영자가 직접 관리하지 않는 것:

- 코드 수정
- 서버 설정 변경
- 데이터 파이프라인 구조 변경
- 배치 로직 수정
- 추천 로직 기준 변경
- DB 직접 수정

위 항목은 개발자 또는 기술 담당자에게 넘겨야 합니다.

## 3. `/admin` 화면을 어떻게 읽어야 하는가

운영자는 숫자를 외우기보다 `어떤 카드가 빨간지`, `어느 탭으로 이동해야 하는지`, `오늘 바로 손봐야 하는지`를 먼저 판단해야 합니다.

### 3-1. `Overview`

`Overview`는 오늘 상태를 여는 첫 화면입니다.

먼저 보는 순서:

1. `전체 상태`
2. `Service Readiness`
3. `공개 전 드라이런`
4. `운영 검증 체크포인트`
5. 상단 경고 요약

각 카드의 의미:

- `전체 상태`
  - 오늘 서비스의 큰 방향입니다.
  - `ok`면 큰 이상이 없다는 뜻입니다.
  - `warning`이면 열 수는 있지만 운영자가 내용을 보고 있어야 합니다.
  - `critical`이면 즉시 원인 확인이 필요합니다.

- `Service Readiness`
  - 서비스가 안정적으로 열려 있는지 보는 카드입니다.
  - 건강 상태, 자동 작업 등록, 일일 배치, 검증 근거, 뉴스 수집, 운영 증빙을 종합합니다.
  - 이 카드가 `blocked`이면 운영자는 다른 작업보다 이 카드를 먼저 해결해야 합니다.

- `공개 전 드라이런`
  - 베타 운영, 소수 사용자 운영, 파일럿 운영을 얼마나 자신 있게 받을 수 있는지 보는 카드입니다.
  - 일상 운영에서는 `지원 흐름`, `피드백 수집`, `운영 관찰 기록`이 끊기지 않는지 보는 참고 카드로 생각하면 됩니다.

- `운영 검증 체크포인트`
  - 실제 운영자가 손으로 확인해야 하는 항목을 기록하는 카드입니다.
  - 스케줄러, 백업, 복구 연습, 롤백 연습, 배포 후 기본 확인이 여기에 들어갑니다.
  - 이 카드는 `운영 증빙 기록`입니다. 실제로 확인하지 않았는데 완료 처리하면 의미가 없습니다.

### 3-2. `Data Quality`

이 탭은 `오늘 추천 결과를 믿어도 되는가`를 보는 곳입니다.

먼저 보는 순서:

1. `validation fallback`
2. `measured validation`
3. `failed batches`
4. `news live fetch`
5. `runtime sync trust`
6. `Fallback recovery and hotspots`
7. `Latest failed steps`

쉽게 해석하면:

- `validation fallback`
  - 직접 근거가 부족해서 보수적으로 대신 계산한 비율입니다.
  - 높을수록 “오늘 추천은 참고는 되지만 확신은 낮다”에 가깝습니다.

- `measured validation`
  - 실제 종목 이력에 가까운 근거 비율입니다.
  - 높을수록 서비스 신뢰가 높습니다.

- `failed batches`
  - 오늘 자동 생성 과정에서 일부 단계가 실패했는지 보여줍니다.
  - 실패 수 자체보다 `어느 단계가 실패했는지`를 같이 봐야 합니다.

- `news live fetch`
  - 뉴스가 실시간에 가깝게 들어왔는지 보는 수치입니다.
  - 낮으면 뉴스 품질 설명이나 공지가 필요할 수 있습니다.

- `runtime sync trust`
  - 운영 문서들이 제때 갱신되고 있는지 보는 신호입니다.
  - `blocked`면 최신 보고서가 비었거나 너무 오래됐다는 뜻입니다.
  - 이 경우 숫자가 좋아 보여도 운영자는 안심하면 안 됩니다.

- `Fallback recovery and hotspots`
  - 어디에서 fallback이 몰렸는지 보여줍니다.
  - 같은 종목의 공용 추적으로 복구된 건수도 여기서 봅니다.
  - 이 화면은 “fallback이 많다”보다 “어디에 집중됐는가”를 읽는 곳입니다.

- `Latest failed steps`
  - 오늘 실패한 구체 단계 이름을 보여줍니다.
  - 운영자는 이 이름을 메모로 남기고 기술 담당자에게 전달하면 됩니다.

### 3-3. `Candidate Ops`

이 탭은 오늘 후보 종목을 운영자가 최종 검토하는 곳입니다.

이 탭에서 하는 일:

- 오늘 후보 종목 수 확인
- 아직 검토하지 않은 종목 수 확인
- 상위 후보에 운영 메모 남기기
- 검토 상태를 바꾸기
- 실제 watchlist에 승격할지 판단하기
- 실패한 배치가 있다면 오늘 후보 신뢰를 낮춰 읽기

운영자가 기억할 원칙:

- 후보를 많이 올리는 것보다 `애매한 후보를 올리지 않는 것`이 더 중요합니다.
- 운영 메모는 나중에 인수인계와 회고의 근거가 됩니다.
- 확신이 없으면 승격하지 않는 편이 낫습니다.

### 3-4. `Users`

이 탭은 고객지원과 계정 관리용입니다.

여기서 할 수 있는 일:

- 사용자 검색
- 세션 수 확인
- 다른 기기 세션 종료
- 이메일/이름/운영 메모 수정
- 일정 기간 계정 정지
- 정지 해제
- 계정 삭제

운영 원칙:

- 사용자가 직접 해결할 수 있는 문제는 먼저 셀프서비스로 안내합니다.
- 세션 종료는 `도용 의심` 또는 `기기 정리 요청`일 때 사용합니다.
- 계정 정지는 명확한 사유가 있을 때만 합니다.
- 계정 삭제는 요청 근거를 남기고 신중하게 처리합니다.

### 3-5. `Notices`

이 탭은 공지 팝업을 관리합니다.

이 탭에서 하는 일:

- 공지 사용 여부 켜기/끄기
- 시작 시각과 종료 시각 설정
- 제목과 본문 작성
- 이미지가 필요하면 추가
- 현재 공지가 실제로 노출 중인지 확인

운영 원칙:

- 공지는 정말 필요한 경우에만 띄웁니다.
- 안내가 길어질수록 사용자는 핵심 작업을 놓치기 쉽습니다.
- 시작/종료 시각을 꼭 넣어서 공지가 끝난 뒤 남아 있지 않게 합니다.

### 3-6. `Portfolio`

이 탭은 운영자가 테스트 포트폴리오나 지원용 자산 상태를 확인할 때 씁니다.

주의할 점:

- 실제 사용자 포트폴리오를 운영자가 직접 고치는 일은 매우 신중해야 합니다.
- 사용자의 기록은 손실, 평단, 리뷰, 규칙까지 연결되므로 가볍게 손대면 안 됩니다.
- 포트폴리오 수정은 반드시 사유를 남기고, 가능하면 개발자와 함께 판단합니다.

## 4. 운영자가 매일 해야 하는 일

이 장은 `오늘 무엇을 해야 하는지`만 빠르게 보고 움직일 수 있게 만든 일일 운영 절차입니다.

### 4-1. 매일 시작 체크

하루를 시작하면 가장 먼저 아래 순서로 확인합니다.

| 순서 | 어디서 본다 | 무엇을 본다 | 정상 기준 | 이상 시 첫 행동 |
| --- | --- | --- | --- | --- |
| 1 | `/admin` `Overview` | 전체 상태 | `ok` 또는 관리 가능한 `warning` | `critical`이면 다른 작업 중단 후 원인 확인 |
| 2 | `/admin` `Overview` | `Service Readiness` | `ready` 또는 이유가 분명한 `monitor` | `blocked`면 서비스 신뢰 문제로 우선 대응 |
| 3 | `/admin` `Overview` | `운영 검증 체크포인트` | 누락 없이 최근 기록 존재 | 오래된 항목은 오늘 확인 일정에 넣기 |
| 4 | `/admin` `Data Quality` | fallback, failed batch, runtime sync | 큰 경고 없음 | 어떤 항목이 붉은지 메모 |
| 5 | 사용자 화면 | `Today`, `Signals`, `Portfolio` | 핵심 화면 정상 렌더 | 깨지면 공지 여부 판단 |

이 단계가 끝나면 운영자는 아래 셋 중 하나로 오늘 상태를 짧게 판단해야 합니다.

- `정상 운영`
- `주의 운영`
- `부분 제한 운영`

### 4-2. 오늘 후보 종목 체크

다음으로 `Candidate Ops`에서 오늘 후보를 봅니다.

이때 꼭 보는 항목:

- 오늘 후보 수
- 아직 검토하지 않은 후보 수
- 상위 후보의 검토 상태
- watchlist 승격 여부
- 실패한 배치가 있었는지

운영 메모를 남겨야 하는 경우:

- 후보를 승격했는데 근거가 애매한 경우
- 후보를 보류했는데 사용자 문의가 들어올 수 있는 경우
- 신뢰도는 낮지만 오늘 꼭 보여야 하는 종목이 있는 경우
- 특정 업종이나 특정 테마에 후보가 과도하게 몰린 경우

### 4-3. 공지 여부 결정

아래 상황이면 공지를 검토합니다.

- 오늘 데이터가 부분적으로 늦는 경우
- 뉴스 수집 비율이 크게 낮은 경우
- 후보 수가 평소보다 지나치게 적은 경우
- 특정 기능이 일시적으로 불안정한 경우
- 휴장일 또는 주말 안내가 필요한 경우

공지의 기준:

- 사용자가 `왜 평소와 다른지`를 바로 이해할 수 있어야 합니다.
- 사용자가 `지금 무엇을 하면 되는지`가 있어야 합니다.
- 불안감을 키우는 표현보다 사실과 행동을 짧게 안내합니다.

### 4-4. 사용자 문의 확인

운영 중에는 최소 하루 한 번 아래를 확인합니다.

- 계정/세션 문제
- 비밀번호 재설정 안내 필요 여부
- 이메일 검증 문의
- 포트폴리오 기록 꼬임 신고
- 튜토리얼/공지/모바일 UI 문의

가능하면 먼저 사용자가 직접 할 수 있는 방법부터 안내합니다.

- 비밀번호 재설정: `Account` 또는 `/reset-password`
- 이메일 검증: `Account`
- 세션 정리: `Account`
- 데이터 내보내기: `Account`
- 계정 삭제: `Account`

직접 처리로 넘어가는 기준:

- 셀프서비스가 정상 동작하지 않는 경우
- 계정 도용 또는 악용이 의심되는 경우
- 기록 상태가 실제로 꼬여 보이는 경우

### 4-5. 하루 마감 체크

하루를 마칠 때는 아래를 확인합니다.

| 항목 | 확인 이유 | 마감 전 해야 하는 일 |
| --- | --- | --- |
| 오늘 경고가 남아 있는지 | 다음 날 반복 문제 방지 | 미해결 항목 메모 남기기 |
| 오늘 공지가 끝났는지 | 오래된 공지 잔존 방지 | 종료 시각 또는 비활성화 점검 |
| 후보 검토가 남아 있는지 | 다음 날 판단 혼선 방지 | 최소 상위 후보는 정리 |
| 사용자 문의가 남아 있는지 | 응답 누락 방지 | 운영 메모 또는 후속 일정 남기기 |
| 운영 검증 기록이 필요한지 | 증빙 공백 방지 | 실제 확인한 항목은 기록 남기기 |

## 5. 매주, 매월, 배포 후 해야 하는 일

### 5-1. 매주 해야 하는 일

- `운영 검증 체크포인트`에서 스케줄러 확인
- 최근 백업 존재 여부 확인
- `Data Quality`에서 fallback이 늘어나는 추세인지 확인
- `Threshold Advice`가 있다면 경향만 읽기
- 오래된 공지 정리
- 정지 상태 사용자와 최근 계정 이슈 점검
- `Today -> Signals -> Portfolio` 기본 흐름 한 번 직접 체험

### 5-2. 매월 해야 하는 일

- 복구 연습 1회
- 롤백 연습 1회
- 배포 후 기본 확인 흐름 재점검
- 모바일 화면 기본 흐름 점검
- 튜토리얼 재생 흐름 점검
- 정책/문의 페이지 링크와 연락처 확인

### 5-3. 배포 후 반드시 해야 하는 일

배포 후에는 아래 순서로 봅니다.

1. 로그인
2. `Today`
3. `Signals`
4. `Analysis`
5. `Portfolio`
6. 공지 팝업
7. 튜토리얼 재실행
8. `/admin` `Overview`

배포 후 이상이 없으면 `운영 검증 체크포인트`의 `배포 스모크`를 기록합니다.

## 6. 운영자가 판단하는 기준

운영자는 숫자를 모두 외울 필요가 없습니다. 아래 기준만 알고 있으면 충분합니다.

### 6-1. 바로 열어도 되는 상태

아래가 동시에 맞으면 `정상 운영`으로 보면 됩니다.

- `Overview` 전체 상태가 `ok`
- `Service Readiness`가 `ready` 또는 사유가 분명한 `monitor`
- `runtime sync trust`가 `healthy`
- 실패한 batch가 없거나 매우 제한적
- 주요 사용자 화면이 정상 렌더

### 6-2. 열 수는 있지만 보고 있어야 하는 상태

아래가 보이면 `주의 운영`입니다.

- 전체 상태가 `warning`
- `validation fallback`이 평소보다 높음
- 일부 batch 실패
- 뉴스 실시간 비율 저하
- 공지로 설명이 필요한 상황

이 경우 운영자는 아래를 해야 합니다.

- 오늘 운영 메모 남기기
- 필요 시 공지 작성
- 사용자 문의 대비 문구 준비
- 같은 문제가 반복되는지 하루 중 한 번 더 보기

### 6-3. 다른 작업보다 먼저 대응해야 하는 상태

아래가 보이면 `부분 제한 운영` 또는 `즉시 대응`입니다.

- 전체 상태가 `critical`
- `Service Readiness`가 `blocked`
- `runtime sync trust`가 `blocked`
- 배치 실패가 핵심 단계에서 발생
- 사용자 핵심 화면이 깨짐
- 기록 저장 또는 undo가 정상 동작하지 않음

이 경우 운영자는 아래 순서로 움직입니다.

1. 문제 범위를 파악합니다.
2. 필요하면 공지부터 올립니다.
3. 사용자 영향이 큰 기능부터 막을지, 유지할지 판단합니다.
4. 메모를 남기고 개발자에게 전달합니다.

## 7. 상황별 대응 가이드

### 7-1. `Service Readiness`가 `blocked`

뜻:

- 오늘 서비스를 믿고 열기 어려운 상태입니다.

운영자가 할 일:

1. `Overview`에서 어떤 체크가 막는지 본다.
2. `Data Quality`로 이동해 문제 원인이 데이터인지 운영 문서인지 확인한다.
3. 사용자 영향이 있으면 공지부터 검토한다.
4. 원인과 시각을 메모로 남긴다.
5. 개발자에게 전달한다.

즉시 같이 남길 메모:

- 언제 확인했는지
- 어떤 카드가 막혔는지
- 사용자 화면까지 영향이 있었는지
- 공지를 띄웠는지

### 7-2. `runtime sync trust`가 `blocked`

뜻:

- 운영 문서가 비었거나 너무 오래돼서, 현재 숫자를 그대로 믿기 어렵습니다.

운영자가 할 일:

1. 어떤 문서가 오래됐는지 본다.
2. 오늘 배치와 자동 복구가 실제로 돌았는지 `Overview`에서 같이 본다.
3. 서비스 자체가 정상처럼 보여도 메모를 남긴다.
4. 같은 증상이 반복되면 개발자에게 넘긴다.

운영자가 기억할 점:

- 이 상태는 `보이는 화면이 멀쩡해도 안심하면 안 되는 상태`입니다.

### 7-3. `failed batches`가 있다

뜻:

- 오늘 자동 생성 과정의 일부가 실패했습니다.

운영자가 할 일:

1. 실패한 단계 이름을 확인한다.
2. 후보 종목 수나 추천 신뢰도에 영향이 있는지 같이 본다.
3. 필요하면 후보 검토를 더 보수적으로 한다.
4. 문제가 반복되면 공지 또는 개발자 전달을 검토한다.

### 7-4. `validation fallback`이 높다

뜻:

- 직접 근거보다 대신 계산한 비율이 높다는 뜻입니다.

운영자가 할 일:

1. `Fallback recovery and hotspots`에서 어느 종목/어느 방식에 집중됐는지 본다.
2. 오늘 상위 후보를 더 보수적으로 읽는다.
3. 사용자가 오해할 수 있으면 공지나 운영 메모를 준비한다.
4. 며칠 연속 높으면 개발자에게 경향을 전달한다.

### 7-5. `news live fetch`가 낮다

뜻:

- 뉴스가 실시간으로 충분히 들어오지 않았을 수 있습니다.

운영자가 할 일:

1. 추천이나 분석 자체보다 뉴스 해석을 보수적으로 읽는다.
2. 필요한 경우 “뉴스 반영이 일부 지연될 수 있다”는 공지를 검토한다.
3. 같은 문제가 반복되면 기술 이슈로 전달한다.

### 7-6. 사용자 포트폴리오가 꼬였다고 문의한다

운영자가 먼저 확인할 것:

- 어떤 종목인지
- 언제 저장했는지
- 어떤 행동 직후 꼬였는지
- `undo`, `reopen`, `부분 익절`, `현금 수정` 중 무엇이 있었는지

운영자가 할 일:

1. 사용자의 설명과 시각을 받는다.
2. 포트폴리오 화면과 저널 흐름을 기준으로 1차 재현 여부를 본다.
3. 직접 수정하기보다 먼저 메모와 증거를 모은다.
4. 실제 데이터 꼬임이 의심되면 개발자에게 넘긴다.

운영자가 바로 수정하지 말아야 하는 이유:

- 포트폴리오는 보유, 현금, 평단, 복기, 규칙까지 연결되어 있기 때문입니다.

### 7-7. 로그인 또는 세션 문제가 있다

운영자가 먼저 안내할 것:

- 비밀번호 재설정
- 다른 기기 로그아웃
- 이메일 검증

운영자가 직접 할 수 있는 것:

- `Users`에서 세션 종료
- 필요 시 계정 상태 점검
- 정지 해제

개발자에게 넘겨야 하는 기준:

- 여러 사용자에게 같은 시각에 반복됨
- 세션이 즉시 끊기거나 로그인 자체가 안 됨
- 셀프서비스가 모두 실패함

### 7-8. 튜토리얼, 공지, 모바일 UI가 겹친다

운영자가 할 일:

1. 어느 화면에서 겹쳤는지 확인한다.
2. 스크린샷과 시각을 받는다.
3. 공지 일정이 원인인지 확인한다.
4. 공지가 긴급하지 않으면 잠시 비활성화 여부를 판단한다.
5. 개발자에게 화면과 재현 조건을 전달한다.

### 7-9. 악성 사용자 또는 운영 정책 위반이 의심된다

운영자가 할 일:

1. `Users`에서 계정 상태와 최근 세션을 본다.
2. 필요한 경우 세션 종료
3. 필요한 경우 일정 기간 정지
4. 운영 메모에 이유 남기기
5. 반복되면 개발자와 정책 논의

## 8. 사용자 문의 대응 기준

운영자가 문의를 받을 때 먼저 받아야 하는 정보는 아래 다섯 가지입니다.

1. 사용자 이메일
2. 문제가 난 화면 이름
3. 문제 발생 시각
4. 기대한 동작
5. 실제로 보인 결과 또는 스크린샷

### 8-1. 문의 유형별 1차 대응

| 문의 유형 | 먼저 안내할 것 | 운영자가 직접 볼 것 | 개발자 전달 기준 |
| --- | --- | --- | --- |
| 로그인 안 됨 | 비밀번호 재설정, 이메일 확인 | 계정 상태, 세션 | 여러 사용자 동시 발생 |
| 다른 기기 로그인 의심 | 다른 기기 로그아웃 | 세션 수, 최근 세션 | 세션이 계속 다시 생김 |
| 포트폴리오 수치 이상 | 저장 직전 행동 확인 | 기록 흐름, 사용자 설명 | 재현 가능 또는 금액 불일치 |
| 추천 신뢰 문의 | 신뢰도 표시 설명 | 오늘 데이터 품질 | 같은 문의가 반복됨 |
| 공지/튜토리얼 불편 | 공지 시각과 위치 확인 | `Notices`, 사용자 화면 | 반복 재현 가능 |

### 8-2. 운영자가 사용자에게 먼저 안내하면 좋은 방향

- 계정 문제는 먼저 `Account`에서 직접 해결 가능한지 안내합니다.
- 포트폴리오 문제는 즉시 수정 약속보다 `정확한 기록 확인 후 안내`를 우선합니다.
- 데이터 지연은 숨기지 말고 짧게 사실만 알립니다.
- 공지나 튜토리얼 이슈는 “현재 확인 중이며, 반복되면 일정 조정 또는 수정하겠다”로 안내합니다.

## 9. 운영자가 반드시 남겨야 하는 기록

운영에서 가장 위험한 것은 `문제보다 기록이 없는 것`입니다.

운영자는 아래 상황에서 반드시 메모를 남겨야 합니다.

- `Overview`에서 큰 경고를 봤을 때
- 공지를 켜거나 끌 때
- 후보 종목을 승격하거나 보류할 때
- 사용자 세션을 종료했을 때
- 사용자 계정을 정지/해제/삭제했을 때
- 포트폴리오 관련 이슈를 접수했을 때
- 운영 검증 체크포인트를 완료했을 때

메모에 꼭 들어가야 하는 요소:

- 언제
- 누가
- 무엇을 보고
- 어떤 판단을 했는지
- 후속 조치가 필요한지

## 10. 운영자가 혼자 처리해도 되는 일

- 공지 등록과 종료
- 오늘 후보의 검토 상태 정리
- watchlist 승격 여부 판단
- 사용자 세션 종료
- 사용자 계정 정지/해제
- 사용자 기본 정보 수정
- 셀프서비스 안내
- 운영 체크포인트 기록

## 11. 운영자가 혼자 결정하면 안 되는 일

- 포트폴리오 숫자를 임의로 수정
- 반복되는 데이터 품질 문제를 숨기기
- 큰 장애가 있는데 공지 없이 유지
- 근거 없이 계정 삭제
- 운영 메모 없이 정지/삭제 처리
- 개발자 확인 없이 데이터 구조 문제를 직접 손보려 하기

## 12. 인수인계 받을 때 꼭 넘겨야 하는 항목

새 운영자에게는 아래가 반드시 전달되어야 합니다.

- `/admin` 접속 방법
- 운영자 토큰 보관 위치
- 지원용 이메일 또는 문의 채널
- 공지 작성 기준
- 사용자 정지/삭제 기준
- 백업과 복구 담당자
- 개발자 또는 기술 담당자 연락 경로
- 최근 한 달 운영 이슈 요약
- 현재 켜져 있는 공지 여부
- 최근 포트폴리오/데이터 품질 이슈 여부

보안 정보는 이 문서 안이 아니라 별도 안전한 문서 또는 비밀 관리 도구에 보관해야 합니다.

## 13. 운영자가 참고하면 좋은 실제 하루 운영 예시

### 오전 첫 확인

1. `/admin` `Overview`
2. `Service Readiness` 확인
3. `Data Quality`에서 오늘 데이터 상태 확인
4. `Today`, `Signals` 화면 직접 확인

### 오전 후반

1. `Candidate Ops`에서 상위 후보 검토
2. 필요하면 공지 작성
3. 문의 채널 확인

### 오후 또는 마감 전

1. 반복 경고가 남았는지 재확인
2. 계정/세션 이슈 정리
3. 오늘 메모 누락 없는지 확인

### 하루 종료 전

1. 남은 위험 요소가 있는지 정리
2. 다음 운영자 또는 다음 날의 자신을 위해 메모 남기기

## 14. 시스템이 운영자의 일일 체크를 대신 볼 수 있는가

결론부터 말하면 `대부분은 가능하고, 일부는 사람이 끝까지 봐야 합니다.`

현재 구조만 봐도 이미 자동으로 보고 있는 항목이 많습니다.

### 14-1. 이미 시스템이 상당 부분 보고 있는 것

현재 시스템은 아래를 이미 자동으로 종합합니다.

- 전체 건강 상태
- 일일 배치 상태
- 자동 복구 상태
- 뉴스 수집 상태
- 추천/분석 생성 상태
- validation fallback 비율
- runtime 문서 신선도
- 운영 증빙 누락 여부
- 드라이런 가능 여부

즉 지금도 `/admin`은 `운영자의 체크리스트를 절반 이상 대신 보고 있는 상태`입니다.

### 14-2. 바로 자동화하기 쉬운 항목

아래는 비교적 빠르게 `시스템 자가 점검`으로 옮길 수 있습니다.

1. `매일 운영 요약 자동 생성`
   - 오늘 상태를 초록/주의/차단으로 자동 요약
   - 운영자가 오늘 꼭 해야 할 일 3개를 자동 정리

2. `운영 체크 누락 알림`
   - 스케줄러 확인이 오래됐는지
   - 백업 확인이 오래됐는지
   - 배포 스모크가 누락됐는지

3. `후보 검토 누락 알림`
   - 상위 후보 중 아직 검토하지 않은 종목 수가 많을 때 알림

4. `공지 만료 점검`
   - 종료 시각이 지난 공지가 아직 살아 있으면 알림

5. `runtime sync 차단 알림`
   - 운영 문서가 오래됐을 때 운영자에게 즉시 알려주기

### 14-3. 개발이 더 필요한 자동화

아래는 가능하지만 한 단계 더 작업이 필요합니다.

1. `실제 사용자 흐름 자동 점검`
   - 테스트 계정으로 로그인
   - `Today -> Signals -> Analysis -> Portfolio`까지 자동 진입
   - 화면 깨짐이나 저장 실패를 자동 감지

2. `포트폴리오 정합성 자동 감시`
   - 저장 직후 정합성 보고서 누적
   - 특정 사용자에게 반복 불일치가 생기면 경고

3. `사용자 지원 자동 분류`
   - 문의 내용을 계정/세션/데이터/기록/UI로 자동 분류
   - 운영자에게 우선순위를 붙여 보여주기

4. `공지 필요 상태 자동 제안`
   - 뉴스 지연, 배치 실패, fallback 급증 시 공지 초안 제안

### 14-4. 사람이 끝까지 봐야 하는 항목

아래는 시스템이 보조는 할 수 있지만, 최종 판단은 사람이 해야 합니다.

- 오늘 공지를 실제로 띄울지 말지
- 어떤 후보를 watchlist로 올릴지
- 사용자에게 어떤 어조로 안내할지
- 포트폴리오 수정이 정말 필요한지
- 계정 정지/삭제가 적절한지

이 영역은 `정책과 판단`의 영역이기 때문입니다.

## 15. 운영자의 일일 체크를 시스템 자가 점검으로 전환하는 권장 단계

### 1단계: 현재 구조를 더 잘 보이게 만들기

목표:

- 운영자가 아침에 `/admin`만 열어도 오늘 해야 할 일을 바로 알게 만들기

권장 작업:

- `오늘의 운영 요약` 카드 추가
- `오늘 해야 할 일` 자동 목록 추가
- `어제와 비교해 나빠진 항목` 자동 표시

### 2단계: 운영자 확인 누락을 자동으로 잡기

목표:

- 운영자가 깜빡해도 시스템이 먼저 알려주게 만들기

권장 작업:

- 오래된 체크포인트 자동 알림
- 공지 만료 알림
- 후보 검토 누락 알림
- support 응답 누락 알림

### 3단계: 실제 사용자 흐름까지 자동 검증하기

목표:

- 운영자가 눈으로 보기 전에 시스템이 먼저 핵심 흐름 이상을 잡기

권장 작업:

- 테스트 계정 기반 자동 로그인 점검
- 핵심 화면 진입 점검
- 공지와 튜토리얼 충돌 점검
- 모바일 주요 화면 기본 렌더 점검

### 4단계: 운영 보조를 넘어 운영 추천으로 가기

목표:

- 시스템이 단순 상태 표시를 넘어 “지금 무엇을 하라”까지 제안

권장 작업:

- 공지 필요 여부 제안
- 사용자 영향 범위 추정
- 후보 검토 우선순위 제안
- 오늘 운영 메모 초안 생성

## 16. 운영자 입장에서 앞으로 더 채워야 할 항목

현재도 운영은 가능하지만, 아래 항목이 더 들어오면 운영 난이도가 크게 낮아집니다.

### 16-1. 가장 먼저 있으면 좋은 것

1. `오늘의 운영 요약`
   - 지금은 운영자가 여러 카드를 보고 스스로 종합해야 합니다.
   - 앞으로는 `오늘 서비스 상태`, `오늘 꼭 할 일`, `사용자 영향 여부`를 한 카드로 모아주는 것이 좋습니다.

2. `운영 메모 보드`
   - 지금은 여러 탭에 메모가 흩어집니다.
   - 오늘의 경고, 오늘 공지 이유, 오늘 후보 판단 이유를 하루 단위로 모아 보는 보드가 있으면 인수인계가 쉬워집니다.

3. `사용자 문의 정리함`
   - 현재는 계정 조치와 상태 확인은 가능하지만, 문의 그 자체를 모아 보는 화면은 없습니다.
   - 문의를 `계정`, `세션`, `기록`, `데이터`, `UI`로 나눠 보는 구조가 있으면 좋습니다.

4. `문제 이력판`
   - 같은 문제가 반복될 때 운영자가 “이건 지난주에도 있었다”를 바로 알아야 합니다.
   - 반복 경고, 반복 공지, 반복 문의를 한 곳에서 보는 보드가 필요합니다.

### 16-2. 운영 품질을 높여 주는 항목

1. `후보 검토 완료 표시`
   - 상위 후보가 모두 검토됐는지, 아직 남았는지를 하루 단위로 더 명확히 보여주면 좋습니다.

2. `공지 히스토리`
   - 어떤 공지를 언제 띄웠고, 왜 내렸는지를 다시 볼 수 있으면 운영 판단이 쉬워집니다.

3. `사용자 영향 범위 표시`
   - 현재 경고가 전체 사용자에게 영향인지, 일부 화면만 영향인지 표시되면 공지 판단이 쉬워집니다.

4. `기록 이슈 전용 경고`
   - 포트폴리오 정합성 이슈는 서비스 신뢰에 직접 연결되므로, 다른 경고와 분리해서 더 강하게 보여주는 편이 좋습니다.

### 16-3. 여러 운영자가 함께 쓸 때 필요한 것

1. `누가 마지막으로 무엇을 처리했는지`
2. `메모 작성자와 시각`
3. `인수인계 체크 상태`
4. `오늘 미해결 항목`

이 네 가지가 있어야 여러 사람이 운영해도 누락이 줄어듭니다.

## 17. 최종 판단

운영자의 일일 체크를 시스템이 상당 부분 대신 보게 만드는 것은 `충분히 가능`합니다.  
이미 `Overview`, `Service Readiness`, `Data Quality`, `운영 검증 체크포인트`, `runtime sync trust` 구조가 있기 때문에, 완전히 새로 만드는 작업이 아니라 `지금 있는 신호를 운영자의 일과 연결하는 작업`에 가깝습니다.

가장 현실적인 방향은 아래입니다.

1. 먼저 지금 있는 신호를 `오늘 해야 할 일` 형태로 묶는다.
2. 그다음 누락과 지연을 자동 알림으로 바꾼다.
3. 마지막에 실제 사용자 흐름 자동 점검을 붙인다.

즉, 운영자의 일을 시스템이 대신 보게 만드는 것은 가능하지만, `공지 판단`, `사용자 응대`, `계정 조치`, `기록 수정 판단` 같은 영역은 끝까지 사람이 책임져야 합니다.

## 19. 마지막 원칙

운영자가 가장 먼저 지켜야 할 원칙은 아래 세 가지입니다.

이 세 가지만 지켜도 서비스 신뢰는 크게 흔들리지 않습니다.
